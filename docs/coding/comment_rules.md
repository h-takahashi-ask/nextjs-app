# コメント記述規約

本リポジトリの**コメントの書き方**を定める規約、新規ファイルおよび既存ファイルへの追加の双方に適用する

---

## 1. 目的と適用範囲

### 1-1. 目的

- 「**他の人が継続開発できる状態**」を作るためのコード可読性向上
- コメント品質を属人化させない
- PR レビュー時の判断基準を統一する

### 1-2. 適用範囲

- 本リポジトリ全体 (`backend/`、`frontend/`、`db/` 配下)
- 既存ファイルへの追加、新規ファイルの両方に適用
- 例外: 自動生成ファイル (例: `package-lock.json`、`pnpm-lock.yaml`) は対象外

---

## 2. 大原則 (8 つ)

### (1) コードのロジックや整形に手を入れない

- コメント追加だけで完結させる
- インデント、改行、空行も極力維持
- 既存のフォーマット違反 (例: 関数途中の `import` 文) を**ついでに**修正しない
- 修正が必要なら**別 PR**で行う

### (2) 嘘は書かない

- コードと一致しない記述は禁止
- 確証が持てない場合は「と推測する」「要確認」と明示する
- 「動作しない」「壊れている」など事実と異なる断定は厳禁

### (3) Why を書く、How はコードに任せる

- 「なぜこの設計か」「なぜこの順序か」「なぜこの値か」だけを書く
- コードを翻訳しただけのコメントは書かない (`// id を返す` など)
- 業務的、歴史的、技術的な背景を残す
- **ライブラリの標準動作の説明は書かない**、React や TanStack Query、SQLAlchemy、FastAPI などの公式仕様は公式ドキュメントに任せる
- 書くのは**このシステム固有の業務的意図、設計判断、既存実装との整合**のみ

### (4) 自明なことは書かない

- `getId()` に「id を返す」と書かない
- `for skill in skills` に「skills をループする」と書かない
- 関数名や変数名で読み取れる情報を繰り返さない

### (5) 古びる情報を含めない

- ❌ 行番号: `// L42 を参照` (リファクタで陳腐化)
- ❌ 可変な日付: `// 2025-01-15 に修正` (履歴は git に任せる)
- ❌ 関係者名: `// 山田さん作` (異動、退職で陳腐化)

### (6) AI 実装である前提を踏まえた実装意図を残す

- 本リポジトリのコードは AI が実装したものを前提とする
- AI が選んだ設計の根拠、代替案との比較で本案に至った経緯、採用ポリシーを設計判断の証跡として残す
- 仕様変更時に「AI が当初なぜそう書いたか」が後から追える状態を維持する
- 既存ロジックの暗黙の前提 (例: 「同一エンジニア内で UNIQUE 制約があるため重複登録は DB 側で弾かれる」) を明文化する

### (7) このシステムを知らない人が読んで理解できる文章にする

- 内輪の略語、暗黙の前提に頼らない
- **ライブラリ / フレームワークの概念用語は、コメント本文では平易な日本語に置き換える** (React 初学者を含む読み手を想定する)
  - 置き換える用語の例: `ミューテーション` → `サーバーのデータを変更する処理 (追加・更新・削除)`、`クエリ` → `サーバーからデータを取得する処理`、`invalidate` → `保持データを古い扱いにして再取得させる`、`ディスパッチ` → `イベントを発火する`
  - **置き換え対象はコメント本文の概念語のみ**、コード上の識別子 (`useMutation` `useQuery` 等の関数名) と 種別マーカー (`【Hook】`) はそのまま使う
  - 識別子を本文で参照する必要がある場合はバッククォートで囲み、動作を平易語で併記する (例: `` `useCreateEngineerSkill` (スキルを追加する処理) ``)
- リポジトリ固有用語のうち、初見で意味が取りにくいものは**主要な使用箇所**で 1 行補足する
  - 補足を入れる例: `ルーター` (FastAPI の `APIRouter`、関連エンドポイント群を束ねる単位)、`フック` (React のカスタムフック、`use*` で始まる関数)
  - 補足を入れない例: `IF` (Interface)、`マスタ` (業務データの分類、候補定義)
  - **随時すべての出現箇所に入れる必要はなく、ファイル冒頭やモジュール先頭などの主要な部分にだけ**入れる
- 「初見の人がたどれる文章」を全コメントで徹底する

### (8) 初学者向けの補足コメントを入れる (本リポジトリ固有ルール)

- 本リポジトリはNext.js / Reactの学習を目的とした初学者向けプロジェクトである
- フレームワーク固有の概念 (ルートレイアウト、App Router、ハイドレーション等) は初見で理解しにくいため、理解に必要な補足コメントを入れる
- 大原則 (3)「Howはコードに任せる」より本原則を優先し、仕組みの背景説明を加えてよい
- 補足の基準: 「Next.js / Reactを初めて触る人がこのコードを読んで、なぜこう書くのかが分かるか」を自問して判断する
  - ✅ 「App RouterではすべてのルートがこのRootLayoutの子として描画される」(仕組みの背景)
  - ❌ 「metadataをエクスポートするとNext.jsがheadタグを生成する」(公式仕様の翻訳に過ぎない)
- 補足が長くなる場合は `@remarks` ブロックにまとめ、インラインコメントに詰め込まない

## 3. 言語

### 3-1. 基本

- **日本語**で書く (既存 docstring に倣う)
- 業務用語は日本語、技術用語は英語のままで OK
- **句読点 (`。` `、`) を改行マーカーや文末記号として機械的に付けない**
  - 対象は **コード内コメント** (Python docstring、TSDoc、SQL コメント、行内コメント) と本規約自身を含む説明文の双方
  - **行末・コメント末尾に句読点を付けない**、改行はそれ自体が文や節の区切りになる、改行位置に句読点は不要
  - **基本は「1 行 = 1 文または 1 節」**にして、文の区切りは改行で表現する
  - **読点 (`、`) は一文の途中の自然な区切りにのみ使う**、二つの文を読点でつないで 1 行に詰め込まない (それぞれ別の行に分ける)
    - 悪い例: `本人のみに限定する、ロール権限による許可は設けない` (二文を読点で接着している)
    - 良い例: 改行して 2 行に分ける → `本人のみに限定する` / `ロール権限による許可は設けない`
  - **句点 (`。`) は原則使わない**、例外として一つの文が長く複数行にまたがる場合のみ、文の終わりを示す目安として句点で区切ってよい
  - 並列の区切りは原則として読点を使い、ファイル名やパスの並列など読点だと意味が紛らわしい場合のみ `/` を使う
- **日本語と英数字の間の半角スペースは、地の文の品詞境界では入れない、構造的に必要な箇所では保持する**
  - 地の文で日本語と英数字が品詞境界として接する場合 (修飾語 + 名詞、名詞 + 助詞等) は **くっつける**
    - 例 (悪) `FastAPI のエンドポイント`、`JWT トークン`、`本 API` で完結する、`` `Depends` で受け取る ``
    - 例 (良) `FastAPIのエンドポイント`、`JWTトークン`、`本API` で完結する、`` `Depends`で受け取る ``
  - 構造的に区切りとして機能している箇所は **保持する** (詰めると可読性や意味解釈が落ちる位置)
    - コロン後の値や Args / Returns のラベルと値の境目 (例: `engineer_id: 更新対象エンジニアの ID 本人以外の更新は ...`)
    - 列挙の区切り (例: `type: 候補の種別 project / customer / engineer のいずれか`)
    - 補足語句 (パーレン外側) と英数字値の境目 (例: `テンプレート種別 main (正) または sub (副)`)
  - 判断基準: 「日本語＋英数字を詰めて読んだとき自然か」、詰めて違和感が出る場合 (`種別main` `IDの本人` 等) はスペースを保持してよい
  - 対象はコメント本文と本規約自身、コード内の文字列リテラルや識別子は対象外 (例: `permission_level = 'ALLOW'` のクオート内はそのまま)

- **例外規則 (記号の前後はスペース制御を区別する)**:

| 記号 | スペース付与 | 例 |
|---|---|---|
| パーレン `( )` | **対象外** (どちらでも可、各ファイルの既存スタイルを尊重) | `スキル(取得)` も `スキル (取得)` も可 |
| コロン `:` | **両側スペース** (構造的区切り) | `MyPage : マイページ画面` |
| 演算子 `/`、`×`、`+`、`-` | **両側スペース** (視認性向上) | `ALLOW / LIMITED / DENY`、`ロール × 機能` |
| マーカー内 `【種別 対象名】` | **保持必須** | `【Page マイページ】` (削ってはならない) |
| 単位や英記号同士 | **従来どおり** | `30 minutes`、`a + b` |
| データ内コロン (時刻、比率) | **両側スペースなし** | `1:1`、`10:30` |

### 3-2. 例外

- 既存ファイルが英語コメントで統一されている場合はそれを尊重する
- ライブラリ仕様への参照は英語表記のまま

| 階層 | 必須レベル | 内容 |

## 5. スタイル別詳細

### 5-0. ファイル冒頭テンプレート (全拡張子共通)

新規ファイルおよび本規約以後のレビュー対象ファイルでは、以下の構造を**全拡張子共通**で適用する

```
【プロジェクト名】【領域】【種別】【対象名】【概要】<役割を 1 行で>
【プロジェクト名】【領域】【種別】【対象名】【概要】<追加の役割説明 (複数行可、各行に同じプレフィックス)>

【プロジェクト名】【領域】【種別】【対象名】【概要】<識別子1> : <業務語彙> (HTTP method URL)
【プロジェクト名】【領域】【種別】【対象名】【概要】<識別子2> : <業務語彙> (HTTP method URL)

【プロジェクト名】【領域】【種別】【対象名】【関連】<コード内パス> : <関連性>
【プロジェクト名】【領域】【種別】【対象名】【関連】<コード内パス> : <関連性>
```

項目は **`【概要】` と `【関連】` の 2 つのみ**  
**対象名はマーカーの一部として各行のプレフィックス内に組み込む**

#### マーカー階層 (5 段)

| 階層 | 例 | 説明 |
|---|---|---|
| プロジェクト名 | `【Next.jsキャッチアップ】` | リポジトリ単位で固定の業務名 |
| 領域 | `【Backend】` `【Frontend】` `【DB】` (`【Infra】` は現時点対象外) | 物理配置の大区分 |
| 種別 | `【API】` `【Hook】` `【Page】` `【Modal】` `【Component】` `【DB Session】` `【API Client】` `【DDL】` `【Seed】` 等 | ファイルの性質 |
| **対象名** | `【エンジニアスキル管理】` `【マイページ】` `【顧客追加】` 等 | ファイルが担う業務の名前 (旧タイトル行の対象名をマーカー化したもの) |
| 項目 | `【概要】` `【関連】` | 内容種別 (この 2 つに固定) |

対象名の決め方:

- backend / frontend で対応するファイル同士は **同じ対象名**を使う (例: `backend/app/api/skills.py` と `frontend/src/api/hooks/useSkills.ts` は両方 `【エンジニアスキル管理】`)
- 「画面」「モーダル」等の接尾辞は付けない (種別マーカーで既に表現されているため)、例: 旧 `マイページ画面` → 新 `【マイページ】`
- 実装手段 (`React Query` `fetch` 等) を含めない、業務語彙のみ

#### 領域 × 種別 マッピング

| 領域 | 種別 | 配置 |
|---|---|---|
| `Backend` | `API` | `backend/app/api/*.py` |
| `Backend` | `Service` | `backend/app/services/*.py` (将来導入) |
| `Backend` | `Core` | `backend/app/core/*.py` (FastAPI / 認証 / 設定の基盤) |
| `Backend` | `Model` | `backend/app/models.py` |
| `Backend` | `Schema` | `backend/app/schemas.py` |
| `Backend` | `AppEntry` | `backend/app/main.py` |
| `Backend` | `DB Session` | `backend/app/db.py` (SQLAlchemy エンジン、`get_db` セッション依存性) |
| `Frontend` | `Hook` | `frontend/src/api/hooks/*`、`frontend/src/hooks/*` |
| `Frontend` | `Page` | `frontend/src/pages/**/*.tsx` のうち `*Modal.tsx` を除くもの (`*Page.tsx` / `*TabContent.tsx` / `*Panel.tsx` / `*Preview.tsx` 等を含む) |
| `Frontend` | `Modal` | `frontend/src/pages/**/*Modal.tsx` |
| `Frontend` | `Component` | `frontend/src/components/**/*.tsx` |
| `Frontend` | `Route` | `frontend/src/routes/*` |
| `Frontend` | `Lib` | `frontend/src/utils/*`、`frontend/src/lib/*` |
| `Frontend` | `Type` | `frontend/src/types/*.ts` |
| `Frontend` | `AppEntry` | `frontend/src/main.tsx`、`frontend/src/App.tsx` |
| `Frontend` | `API Client` | `frontend/src/api/client.ts` (axios インスタンス、JWT 自動付与) |
| `DB` | `DDL` | `db/init/01_schema.sql` (冒頭コメントのみ対象、テーブル / 列単位のコメントは現時点対象外) |
| `DB` | `Seed` | `db/init/02_seed.sql` (冒頭コメントのみ対象) |

新しい種別を追加する場合は本表に追記する、別表現の混在 (例: `Hooks` `Page Component`) は禁止する

**現時点のコメント対象外**:

- `db/init/*.sql` の各テーブル単位のコメント (テーブル区切り罫線・列コメント・MySQL `COMMENT` 句) — 冒頭コメントのみ対象
- `db/migrations/*.sql` 配下のファイル全般
- `docker-compose.yml` (Infra Compose)
- `*/Dockerfile` (Infra Dockerfile 類)

> 適用対象ファイル数の集計は [comment_target_files.md](./comment_target_files.md) を参照

#### 項目の中身

##### 【概要】の書き方

1. **役割説明** (1 行〜複数行)
   - ファイル全体が何をするかを 1 行に 1 トピックで分割して書く、長文を 1 行に詰め込まない
   - 複数行になる場合は**各行に同じプレフィックス** (`【プロジェクト名】【領域】【種別】【対象名】【概要】`) を付与する

2. **URL を持つ要素の列挙** (任意、上記の役割説明とは空行で区切る)
   - 形式: `<識別子> : <業務語彙> (HTTP method URL)`、コロン両側スペース
   - **URL を持たないメソッド・型・内部 helper は記載不要**、列挙対象は API エンドポイントや Hook の通信メソッド等の URL 起点のみ
   - 業務語彙は backend / frontend で揃える (例: 「スキル名候補マスタ取得」を双方で使用)
   - 整列のため**半角スペース** ` ` を連続で使って `:` や `(` の縦位置を揃える (任意、可読性向上のため)、全角スペース `　` は使わない (フォントによって表示幅が変わりインデントが揃わないため)

##### 【関連】の書き方

- 形式: `<コード内パス> : <関連性>`、各行 1 ファイル
- 関連が複数ある場合は**各行に同じプレフィックス**を付与して列挙
- 設計ドキュメント (`docs/` 配下) への参照は書かない
- 行番号、可変な日付、関係者名は含めない

#### 意図

- 各行に `【プロジェクト名】【領域】【種別】【対象名】【項目】` を冗長に付与する理由は、`grep` ヒット行単体で「どのプロジェクトのどの領域のどの種別のどの業務の何の内容か」が分かるようにするため
- 階層化することで `grep "【Backend】【API】"` (全 API ファイル) `grep "【エンジニアスキル管理】"` (backend / frontend ペア両方) のような縦横の絞り込み検索が可能になる
- マーカー語彙は固定する、別表現 (`Description` `Purpose:` 等の英語表現や `役割:` 等の和文) は混在させない

#### 禁止事項

- `docs/` 配下のドキュメント参照は書かない (ファイル移動 / 削除で陳腐化しやすいため)、ファイルパス参照はコード内に留める
- 行番号、可変な日付、関係者名は含めない (大原則 5)
- URL を持たない識別子 (内部 helper、private 関数、型定義のみ等) を 【概要】 に列挙してはならない、概要は **役割説明 + URL を持つ要素のみ**
- マーカー内のスペース (`【プロジェクト名】` 等の文字並び) を削除してはならない、grep スクリプトや一括置換でマーカー内を壊さないこと
- **5 段マーカーの一部欠落** (例: 対象名マーカーを抜かす) は禁止、必ず 5 段揃える

#### 標準サンプル (Backend API)

```python
"""
【Next.jsキャッチアップ】【Backend】【API】【エンジニアスキル管理】【概要】エンジニア個人の保有スキル(``engineer_skills``)のCRUD
【Next.jsキャッチアップ】【Backend】【API】【エンジニアスキル管理】【概要】関連マスタ(``skill_categories``、スキル名候補``skill-master``)の取得

【Next.jsキャッチアップ】【Backend】【API】【エンジニアスキル管理】【概要】list_skill_master     : スキル名候補マスタ取得        (GET    /api/v1/skill-master)
【Next.jsキャッチアップ】【Backend】【API】【エンジニアスキル管理】【概要】list_skill_categories : スキルカテゴリ一覧取得        (GET    /api/v1/skill-categories)

**ポイント**:

- 各行に **5 段マーカー** (`【Next.jsキャッチアップ】【Backend】【API】【対象名】【項目】`) を冗長付与する、grep ヒット行単体で「どのプロジェクトのどの領域のどの種別のどの業務の何の内容か」が判別できる

- 役割説明と URL 列挙の間、URL 列挙と【関連】の間は **空行 1 行** で区切る

 */
```

SQL の DDL は URL を持たないため、**【概要】は役割説明のみ**、テーブル一覧の列挙は不要 (各テーブルの定義直前にテーブル区切り罫線で個別に表現する)
関連リンクは**コード内のファイルパス**のみを書き、設計ドキュメント (`docs/` 配下) への参照は書かない
設計書とコードの整合性は別タスクで担保する性質の問題で、コード内コメントが情報源になると古びるリスクが高いため

#### 列コメントは「列の上に独立行」で書く

行末コメント (`-- ...`) は行が長くなり読みにくいため避ける
列定義の**直前の行**にコメントを置き、コメントとコードを縦方向に揃える

```sql
CREATE TABLE `engineer_skills` (
  -- 主キー
  `id` INT UNSIGNED NOT NULL AUTO_INCREMENT,
  -- 所属エンジニア、engineer 削除時にカスケードで本レコードも削除される
  `engineer_id` INT UNSIGNED NOT NULL,
  -- スキル分類 (言語、FW、DB 等)、skill_categories を参照
  `category_id` INT UNSIGNED NOT NULL,
  -- スキル名、同一エンジニア内で重複禁止 (UNIQUE)、大文字小文字はアプリ層で正規化する
  `name` VARCHAR(100) NOT NULL,
  ...
);
```

#### `COMMENT` 句の使用 (将来方針 / 本タスクは対象外)

MySQL は `CREATE TABLE` の **テーブル単位、列単位** で `COMMENT` 句を持てる
将来的にはこの句を使い、`SHOW CREATE TABLE` や DB ツール (DBeaver 等) からも参照可能なメタデータとしてコメントを保持することを推奨する

将来形 (参考):

```sql
-- ==================================================================================================================================================
-- == engineer_skills (エンジニア個人で登録するスキル)
-- ==================================================================================================================================================

CREATE TABLE `engineer_skills` (
  -- 主キー
  `id` INT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主キー',
  -- 所属エンジニア、engineer 削除時にカスケードで本レコードも削除される
  `engineer_id` INT UNSIGNED NOT NULL COMMENT '所属エンジニア',
  ...
  PRIMARY KEY (`id`)
) ENGINE=InnoDB COMMENT='エンジニア個人で登録するスキル';
```

ただし**本タスクでは対象外**、`COMMENT` 句の追加は DDL 構造変更を伴うため別 PR で計画的に実施する
本タスクで追加するのは `--` の上行コメントと `/* */` のブロックコメントのみ

#### インデント

- 列定義は `2 スペース` で揃える
- コメント開始位置 (`-- `) を列定義と同じインデントに合わせる
- カンマやコメント末尾を縦方向に揃えようとすると保守コストが高くなるため、**揃えるのは行頭インデントのみ**にする

### 5-4. JSX インラインコメント (TSX)

#### 構文の使い分け

TSX ファイルはコメントを書く位置によって使える構文が異なる

| 書く位置 | 使う構文 | 備考 |
|---|---|---|
| JSX マークアップの内側 | `{/* コメント */}` | JSX パーサーが `//` を解釈しないため必須 |
| JavaScript スコープ (関数本体・`.map()` コールバックの `return` より前等) | `//` または `/* */` | 通常の JavaScript 構文が使える |

```tsx
// ✅ JavaScript スコープ: // が使える
const isActive = pathname === link.href;

return (
  <div>
    {/* ✅ JSX マークアップ内: {/* */} を使う */}
    <p>テキスト</p>
  </div>
);
```

#### 書くべき内容

大原則 3・8 に従い、コードを読めば分かる「何」ではなく「なぜ」を書く

- UI 構造の「なぜ」: 条件付きレンダリングの意図、特定のラッパー要素が必要な理由
- 初学者向け補足 (大原則 8): Next.js / React 固有の動作で初見に分かりにくい箇所
- 切り替え実験コード: 代替実装を並べて挙動を比較できるようにする (学習目的の用途)

#### 書かない内容

- 自明な構造: `{/* ヘッダー */}` `{/* ボタン */}` などコードを読めば分かるもの
- ライブラリの標準動作の翻訳 (大原則 3)

#### 切り替え実験コメントの形式 (学習サンプル固有)

実装 A と実装 B を切り替えて挙動を比較したい場合の標準形式

説明コメントは `//`、無効化した代替コードのブロックは `/* */` で囲む (JavaScript スコープなので `//` が使える)

```tsx
// ── [概念A] と [概念B] の違いを体験する ─────────────────────────────────────
// 【現在の実装】挙動の説明
//
// 【代替実装】挙動の説明
//
// ▼ 切り替え手順: 具体的な操作手順
// ─────────────────────────────────────────────────────────────────────────────
return (
  <現在の実装 />
);
/* ── 代替実装 (上の return をコメントアウトしてから下のコメントを外す) ─────
// return (
//   <代替実装 />
// );
──────────────────────────────────────────────────────────────────────────── */
```

> 実例: `src/components/dashboard/nav-links.tsx` — `<Link>` vs `<a>` の挙動比較
