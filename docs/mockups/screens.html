ポイント

  • Edge ランタイムとは — Vercel の CDN エッジサーバーで動く軽量な実行環境。通常の Node.js サーバーとは異なり、世界中の CDN ノードで並列実行されるためユーザーに近い場所で高速に動く。ただし Node.js 専用の機能 (bcrypt / Prisma / ファイルシステム) は使えない
  • proxy.ts はページ描画前に実行される — src/ 直下に置くとすべてのリクエストに対して自動実行される。Edge ランタイムで動くため、Node.js 専用モジュールを含まない auth.config.ts のみ import する
  • layout.tsx は入れ子になる — RootLayout → DashboardLayout の順に包まれ、children に各 page が入る
  • error.tsx は CC 必須 — React のエラー境界は componentDidCatch を使うためクライアントコンポーネントでなければならない
  • (overview) / (site) はルートグループ — URLには影響せず、フォルダ整理のためだけに使う括弧記法
  • loading.tsx vs 手動 Suspense — loading.tsx はページ全体を1つの Suspense で包むショートカット。コンポーネント単位で制御したい場合は手動 Suspense を使う

ポイント

  • A / B の URL 更新は replace() で書き換えるため「戻る」ボタンで検索/ページ履歴が積まれない
  • Suspense key={query + currentPage} — URL が変わるたびに Suspense をリセットし Skeleton を再表示させる
  • redirect() は try の外 — redirect() は内部でエラーをthrowする仕組みのため try の中に置くと catch に捕まりリダイレクトが無効になる
  • useActionState の prevState — Server Action の第1引数に必ず渡される直前の state。関数内で使わなくても引数として宣言が必要

独立ストリーミングの恩恵

/dashboard には 3つの Suspense がある。CardWrapper が遅くても RevenueChart / LatestInvoices は先に表示される。1つの await で全部待つより体感速度が向上。
key={query + currentPage} の意味

URL が変わるたびに key が変わり → Suspense がリセットされ → Skeleton が再表示される。key がないとデータが突然切り替わり、ローディング表示が出ない。

Skeleton の場所

  • Skeleton コンポーネントはすべて src/components/common/skeletons.tsx に集約
  • Skeleton はデータなしで即座にレンダリングできるため、ページの「大枠」を先に見せてレイアウトシフトを防ぐ役割もある
④ URL パラメータ影響マップ どのコンポーネントが何を読み書きするか ▼

URL が State 代わりになる理由

検索語やページ番号を React の state ではなく URL に保存することで…
・ブラウザの「戻る/進む」でページ移動の履歴が保持される
・URL を共有すると同じ検索結果が再現できる
・サーバーコンポーネントが直接 URL を読める
useSearchParams と searchParams の違い

useSearchParams() — Client Component 用フック
 → Pagination / Search が使用

props.searchParams — Server Component 用 props
 → page.tsx が使用 (Next.js 15 で Promise 化)

Search が query を「初期値読み」する理由

  • URL を直接開いた時 / リロード時に入力欄に検索語を復元するため
  • defaultValue={searchParams.get('query')} で非制御入力として読み込む
  • value= (制御入力) にすると入力のたびに state 更新が走るため defaultValue を選択

🖥️ Server Component [SC]
  • サーバー側で実行される
  • DB・ファイルシステムに直接アクセス可
  • JSはブラウザに送られない → バンドルサイズ削減
  • SEO・初期表示に有利
  • 'use client' 宣言は不要
  • App Router のデフォルト
🌐 Client Component [CC]
  • ブラウザで実行される
  • useState / useEffect などフックが使える
  • onClick などイベントハンドラが使える
  • localStorage などブラウザAPIが使える
  • ファイル先頭に 'use client' が必要
  • JSがブラウザに送られる → バンドルに含まれる

機能・特徴 SC CC 補足

useActionState の動作フロー

  • ① 初回表示: state = initialState ({ message: null, errors: {} }) → エラーなし
  • ② フォーム送信 → createInvoice(prevState, formData) サーバーで実行
  • ③ バリデーション失敗 → return { errors: { customerId: ['...'] } }
  • ④ useActionState が戻り値を state として保持
  • ⑤ state.errors?.customerId を参照してエラーメッセージ表示 (再レンダリング)
  • ⑥ バリデーション成功 → DB保存 → redirect() で一覧へ遷移 (stateは更新されない)

create-form と edit-form の主な違い

  • 初期値: create は空欄 / edit は defaultValue={invoice.xxx} でDB現在値を表示
  • Server Action: create は createInvoice / edit は updateInvoice.bind(null, invoice.id)
  • bind パターン: updateInvoice(id, prevState, formData) の id を bind で固定。フォームから formData だけが渡されても id は束縛済み
  • date フィールド: edit では意図的に更新しない (作成日を保持)
  • 404対応: notFound() → not-found.tsx が表示 (create にはない)
